# Telegram-бот «Заявки в отдел маркетинга» — КНУС

Бот: [@knus_marketing_bot](https://t.me/knus_marketing_bot)

Принимает заявки от департаментов университета через диалог с кнопками, нормализует описание
через LLM (Qwen3 на OVMS), создаёт задачу в Microsoft Planner и присылает карточку заявки
в чат отдела маркетинга. Работает по long polling — вебхук и открытые порты не нужны.

## Структура

```
main.py              точка входа, сборка Application, long polling
bot/config.py        чтение .env
bot/constants.py     справочники: департаменты, типы задач, приоритеты
bot/handlers.py      ConversationHandler — вся машина состояний
bot/ticket.py        модель заявки, номер, форматирование текстов
bot/llm.py           нормализация описания (OpenAI-совместимый эндпоинт)
bot/graph.py         Microsoft Graph: Planner (задачи) + SharePoint (файлы)
bot/db.py            SQLite: связь пользователь→заявки для «Мои заявки»
bot/storage.py       jsonl-логи заявок и fallback
```

## Сценарий

`/start` → департамент → тип задачи → описание → **материалы** → дедлайн (`ДД.ММ.ГГГГ` или
«Без срока») → приоритет → контакт → сводка → ✅ Отправить / ✏️ Исправить / ❌ Отмена.

«Исправить» открывает меню полей и возвращает к нужному шагу, после правки — сразу обратно к сводке.
`/cancel` сбрасывает диалог в любой момент, `/start` начинает заново, `/my` — статус своих заявок.

## Переменные .env

Скопируй `.env.example` в `.env` и заполни:

| Переменная | Что это | Статус |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | Токен от @BotFather | обязательно |
| `MARKETING_CHAT_ID` | Куда слать уведомления о заявках | обязательно, см. ниже |
| `MAX_ATTACHMENTS` | Лимит файлов на заявку | `10` |
| `GRAPH_TENANT_ID` / `GRAPH_CLIENT_ID` / `GRAPH_CLIENT_SECRET` | Регистрация приложения в Entra | обязательно |
| `PLANNER_PLAN_ID` | ID плана Planner | обязательно |
| `PLANNER_BUCKET_ID` | Bucket, куда падают заявки | id сегмента «📥 Новые заявки» |
| `PLANNER_BUCKET_NAME` | Запасной поиск bucket'а по имени, если id пуст | `📥 Новые заявки` |
| `PLANNER_ENABLED` | `false` — заявки только в jsonl, без Planner | `true` |
| `LLM_BASE_URL` / `LLM_MODEL` / `LLM_API_KEY` / `LLM_TIMEOUT` | Qwen3 на OVMS | обязательно |
| `LLM_NORMALIZE` | Включить нормализацию описания | `true` |
| `PLATFORM_NETWORK` | Имя внешней docker-сети платформы | `platform_net` — **проверь имя на сервере** |
| `DATA_DIR` | Каталог jsonl-логов внутри контейнера | `/data` |
| `REQUESTS_DB_PATH` | SQLite «Мои заявки» — обязательно на volume | `/data/requests.db` |
| `LOG_LEVEL` | Уровень логов | `INFO` |

`.env` в `.gitignore` — секреты в репозиторий не попадают.

## Как получить MARKETING_CHAT_ID

Telegram API принимает для лички только **числовой** id — `@username` не сработает.

1. Запусти бота (локально или в контейнере).
2. Открой в Telegram [@knus_marketing_bot](https://t.me/knus_marketing_bot), нажми **Start**.
3. Отправь `/myid` — бот ответит `chat_id: 123456789`.
4. Подставь это число в `MARKETING_CHAT_ID` в `.env` и перезапусти бота.

Для группы: добавь бота в группу, отправь `/myid` там — id группы будет отрицательным
(например `-1001234567890`). Бот должен быть участником группы, иначе отправка упадёт.

## Локальная проверка (без Planner)

Режим, где заявка нигде не создаётся, а только пишется в jsonl:

```bash
python -m venv .venv
.venv/Scripts/pip install -r requirements.txt   # Linux: .venv/bin/pip

# в .env:
#   PLANNER_ENABLED=false
#   LLM_NORMALIZE=false   # если LLM-сервер недоступен с твоей машины
#   DATA_DIR=./data

.venv/Scripts/python main.py                    # Linux: .venv/bin/python main.py
```

Пройди диалог в Telegram — заявка ляжет в `./data/requests.jsonl` со статусом `local_only`,
уведомление всё равно уйдёт в `MARKETING_CHAT_ID` (с пометкой, что в Planner не сохранено).

С `PLANNER_ENABLED=true` заявки создаются в реальном плане — для тестов лучше `false`.

LLM и Planner недоступны с твоей машины? Ничего страшного: недоступный LLM просто пропускает
нормализацию, недоступный Planner — пишет заявку в `data/planner_fallback.jsonl` и говорит
пользователю «временно не сохранена».

## Деплой контейнера

На сервере, в общей сети платформы:

```bash
# 1. Проверь имя внешней сети и впиши его в PLATFORM_NETWORK в .env
docker network ls

# 2. Заполни .env (главное — MARKETING_CHAT_ID)

# 3. Сборка и запуск
docker compose up -d --build

# 4. Логи
docker compose logs -f marketing-bot
```

В логах при успешном старте: `Бот запущен: @knus_marketing_bot (id ...)` и `Запуск long polling…`.

Обновление:

```bash
git pull && docker compose up -d --build
```

Порты наружу не публикуются, вебхук не используется — контейнеру нужен только исходящий доступ
к `api.telegram.org`, `login.microsoftonline.com`, `graph.microsoft.com` и к LLM-серверу
во внутренней сети платформы.

## Логи заявок

`./data` смонтирован в контейнер как `/data`:

- `requests.jsonl` — **каждая** отправленная заявка, независимо от Planner
  (`status`: `planner_ok` / `planner_failed` / `local_only`);
- `planner_fallback.jsonl` — только заявки, которые не удалось создать в Planner, с текстом ошибки.

## «Мои заявки»

Кнопка «📋 Мои заявки» на `/start` и команда `/my`. Список — inline-кнопки вида
`№MKT-… · Дизайн · краткая суть…`, по 8 на страницу с навигацией «⬅️/➡️». По нажатию бот
читает задачу из Planner **в момент просмотра** (не кеширует — маркетолог мог только что
подвинуть карточку). Кнопка «⬅️ К списку» возвращает обратно.

### Откуда берётся статус

Статус определяется **сегментом**, в котором лежит карточка — доска и есть поток работы:

| Сегмент | Статус для заявителя |
|---|---|
| 📥 Новые заявки | 🆕 Принята, ждёт очереди |
| ✅ Принято в работу | ✅ Принята в работу |
| 🎨 В производстве | 🎨 В работе |
| 👀 На согласовании | 👀 На согласовании |
| ⏸ Ожидает уточнений | ⏸ Нужны уточнения — с вами свяжутся |
| 🚀 Готово | ✅ Готово |

Отметка «Завершено» на карточке (`percentComplete=100`) перекрывает сегмент: её ставят, не
перетаскивая задачу. Если сегмент неизвестен (переименовали, добавили новый) — заявитель увидит
имя сегмента как есть; если и его нет — запасной путь по `percentComplete`
(`0` → 🆕 Новая, `1–99` → 🔧 В работе, `100` → ✅ Готово).

Почему не по `percentComplete`: он даёт всего три состояния, а доска — шесть, включая
«На согласовании» и «Нужны уточнения», которые в проценте не выражаются. Соответствие
«сегмент → статус» — в `BUCKET_STATUS` ([constants.py](bot/constants.py)); при переименовании
сегментов правь там. Имена сегментов кешируются на 5 минут, сам статус — никогда.

Раздел живёт вне `ConversationHandler`, поэтому `/my` работает и посреди заполнения заявки,
не сбивая её.

Если задачу удалили из плана или Graph ответил ошибкой — «❔ Статус недоступен — уточните
у отдела маркетинга». Заявка при этом остаётся в списке.

**Где хранится связь.** SQLite `/data/requests.db` (`REQUESTS_DB_PATH`) на том же volume, что
и jsonl-логи, — переживает `docker compose up --build`. Строка пишется при **успешном** создании
задачи в Planner: без `planner_task_id` статус читать неоткуда. Заявку отдаём только её автору —
`id` из `callback_data` проверяется вместе с `telegram_user_id`.

**Старые заявки не появятся.** База начинает наполняться с момента внедрения; заявки, поданные
раньше, в ней отсутствуют и в списке не показываются. Они никуда не делись — лежат в
`requests.jsonl` и в Planner.

## Материалы к заявке (гибридное хранение)

Один файл — два маршрута, и скачивается он **ровно один раз**:

1. **Чат маркетинга** — файл пересылается по `file_id` (`send_photo`/`send_document`/…).
   Скачивания нет вообще, байты через бота не идут, поэтому мгновенно.
2. **SharePoint + Planner** — файл скачивается один раз через `getFile` и загружается в библиотеку
   группы плана, а ссылка прикрепляется к задаче как reference. Временный файл удаляется сразу
   после загрузки.

Порядок в `_submit`: создать задачу → ответить заявителю → карточка и файлы в чат → загрузка
в SharePoint → однократная запись `details` (описание + ссылки). Заявитель и маркетинг не ждут
SharePoint, а `details` пишется один раз, когда результат загрузки уже известен.

**Где файлы:** библиотека группы плана → `Заявки маркетинга/<номер заявки>/<имя файла>`.
Материалы одной заявки лежат в своей папке. Одноимённые файлы не затирают друг друга
(`conflictBehavior=rename`). Файлы >4 МБ грузятся через upload session кусками по 3.2 МБ,
чтобы не держать всё в памяти.

**Большой файл (>20 МБ).** Bot API не отдаёт такие через `getFile`. Заявка не падает: файл
принимается, уходит в чат маркетинга по `file_id` (это работает при любом размере), а бот
предупреждает, что в общую папку файл не попадёт и его лучше прислать ссылкой. В задаче Planner
такой файл помечен «слишком большой для загрузки — только в чате маркетинга».

**Сбой SharePoint.** Задача в Planner создаётся, файлы в чате уже есть, а в описании появляется
«материалы в чате маркетинга (SharePoint недоступен)». Ошибка Graph пишется в лог; при 401/403
подсказка про `Sites.ReadWrite.All` + admin consent. В jsonl-логи попадают имена, типы, размеры
вложений и статус загрузки каждого.

## Устройство плана Planner

Сегменты (слева направо) — поток заявки:

`📥 Новые заявки` → `✅ Принято в работу` → `🎨 В производстве` → `👀 На согласовании` →
`⏸ Ожидает уточнений` → `🚀 Готово`

Бот кладёт заявки в `📥 Новые заявки` и сам проставляет метки: тип задачи (`🎨 Дизайн`,
`🎬 Видео`, `📱 Соцсети`, `🖨 Полиграфия`, `🎪 Мероприятие`) и `🔴 Срочно` при высоком приоритете.
Тип «Другое» метки не получает. Соответствие «тип → метка» — в `TASK_TYPE_CATEGORY`
([constants.py](bot/constants.py)); если менять метки в Planner, правь и там.

## Про выбор bucket'а

Graph отдаёт bucket'ы **не** в том порядке, в котором они видны в интерфейсе Planner: первым
в ответе идёт «Завершено», и первая тестовая заявка улетела именно туда. Поэтому bucket
выбирается так, по убыванию приоритета:

1. `PLANNER_BUCKET_ID`, если задан (сейчас — «Новые задачи»);
2. поиск по имени из `PLANNER_BUCKET_NAME`;
3. первый bucket плана — только как последний резерв, с `WARNING` в логе.

## Что проверено на живых данных

- Полный проход диалога в Telegram, заявка `MKT-20260716-2FCT` создана в Planner
  (`[Дизайн] хочу как в кино — ДИТ`): приоритет, дедлайн и `details.description` записались.
- Токен Graph по client credentials, чтение buckets, создание задачи, перенос между bucket'ами.
- SharePoint: определение группы плана по `owner` плана, загрузка файла в
  `Заявки маркетинга/<заявка>/`, прикрепление ссылки к задаче как reference и чтение её обратно.
  Тестовые артефакты (папка `_SELFTEST`, временный reference) удалены после проверки.
- Уведомление в личный чат из `MARKETING_CHAT_ID`.
- Недоступный LLM корректно деградирует: нормализация пропускается, заявка не блокируется.
- Разрыв соединения с `api.telegram.org` (`httpx.ReadError`) — PTB сам переподключает polling.
- Запись обоих jsonl-логов.
- «Мои заявки»: статус по сегменту — прогнаны все шесть сегментов живого плана, отметка
  «Завершено» поверх сегмента, запасной путь по `percentComplete` и удалённая задача
  (`404 → None` → «статус недоступен»).
- SQLite: порядок «новые сверху», отсутствие дублей при повторной вставке, изоляция —
  чужие заявки не видны ни в списке, ни по прямому `id`; пагинация 8 на страницу.

Нормализация через LLM на живом сервере пока не проверялась — он доступен только
из сети платформы. Проверить после деплоя контейнера: в логах не должно быть
`LLM недоступен`, а в сводке должна появиться строка «Кратко (LLM)».
